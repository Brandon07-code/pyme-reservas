                      Introducción            Cualquier negocio maneja información. El reto no está en tenerla, sino en administrarla de manera organizada. En muchos establecimientos pequeños de Cartago, Valle del Cauca, esta tarea todavía se realiza con agendas físicas, cuadernos o conversaciones por WhatsApp. Mientras la cantidad de clientes es reducida, este método suele funcionar sin mayores inconvenientes. Sin embargo, cuando la demanda aumenta empiezan a aparecer problemas como citas duplicadas, pérdida de información o dificultades para consultar el historial de atención de cada cliente.           Ese era el caso de la Barbería y Perfumería      JyM     , ubicada en Cartago. Las citas se coordinaban por WhatsApp y luego se registraban de forma manual. Durante los primeros meses este procedimiento funcionó, pero el crecimiento del negocio empezó a evidenciar algunas limitaciones. Se presentaban cruces de horarios, era complicado consultar los servicios realizados a cada cliente y no existía un registro organizado de la información necesaria para el funcionamiento diario del establecimiento. A partir de esa situación surgió este proyecto.           Para atender esta necesidad se desarrolló una aplicación web que centralizó la gestión de citas, clientes, empleados y servicios en una sola plataforma. Para su construcción se utilizó Laravel como      framework      de desarrollo, PHP para la lógica del servidor, MySQL como sistema gestor de bases de datos y Docker para mantener un entorno de desarrollo estable y fácil de replicar. También se emplearon Blade para desarrollar las vistas del sistema,      Eloquent      para facilitar el acceso a la base de datos mediante el modelo objeto-relacional y el patrón de arquitectura MVC para organizar los diferentes componentes de la aplicación.           El desarrollo se llevó a cabo mediante la metodología RAD, ya que permitió construir el sistema de forma     progresiva    . Cada etapa incluía el desarrollo de nuevas funcionalidades, su validación y los ajustes necesarios antes de continuar con la siguiente. Este enfoque de trabajo      facilitó corregir errores durante el proceso e incorporar mejoras a medida que surgían nuevas necesidades identificadas en el negocio. Finalmente, se incluyen las conclusiones, las recomendaciones y las referencias bibliográficas        .                   Justificación                  La idea de desarrollar este proyecto surgió al identificar las dificultades que presentaba la Barbería y Perfumería          JyM          al momento de administrar sus servicios y reservas. Las citas se coordinaban por WhatsApp y luego se registraban de forma manual, mientras que la información de clientes, empleados y servicios permanecía distribuida entre conversaciones y anotaciones en papel. Este procedimiento funcionó mientras la cantidad de clientes fue reducida; sin embargo, el crecimiento del establecimiento hizo más frecuentes situaciones como horarios duplicados, dificultades para consultar el historial de atención y una organización poco eficiente de la información.                 Aunque actualmente existen diferentes aplicaciones para gestionar reservas, muchas están dirigidas a empresas con necesidades distintas o incorporan funciones que pequeños establecimientos no requieren. Del mismo modo, todavía es común encontrar negocios que continúan administrando sus procesos de forma manual porque las herramientas disponibles no se ajustan a su manera de trabajar. Por esta razón, el sistema se diseñó pensando en las necesidades          específicas del caso de estudio, priorizando las funcionalidades que realmente aportan al funcionamiento diario del establecimiento.                 La aplicación permitió reunir en un solo lugar la información relacionada con clientes, empleados, servicios, reservas e inventario. Disponer de estos datos de forma centralizada facilitó el seguimiento de las actividades, redujo la dependencia de registros físicos y agilizó la consulta de la información cuando fue necesario. Aunque el desarrollo tomó como referencia la Barbería y Perfumería          JyM         , la solución puede servir como base para otros establecimientos del sector servicios que trabajen mediante citas o reservas y cuenten con procesos similares.                 Además del beneficio para el establecimiento, este trabajo representó una oportunidad para fortalecer la formación profesional. Durante su desarrollo fue necesario analizar una situación real, definir los requerimientos del sistema, diseñar la base de datos, implementar la lógica de negocio y construir una aplicación web funcional. Cada una de estas actividades permitió poner en práctica conocimientos adquiridos a lo largo de la carrera y comprender cómo se integran diferentes áreas de la Ingeniería de Sistemas para resolver necesidades concretas.                 Finalmente, el desarrollo fue viable gracias al uso de tecnologías ampliamente empleadas en el desarrollo de aplicaciones web, entre ellas Laravel, PHP, MySQL y Docker. Estas herramientas facilitaron la organización del proyecto y permitieron trabajar en un entorno estable durante todo el proceso. Marco Referencial                   Marco Teórico                      Metodologías de Desarrollo de Software                Desarrollar software implica mucho más que escribir código. Antes de comenzar es necesario entender el problema que se pretende resolver, definir los requerimientos, organizar las actividades y establecer una forma de trabajo que permita avanzar de manera ordenada. Las metodologías de desarrollo surgieron precisamente para responder a esa necesidad. Más que un conjunto de reglas, representan una guía que ayuda a planificar, construir, probar y mantener un producto de software siguiendo un proceso estructurado (Pressman y Maxim, 2021).                No existe una metodología que funcione para todos los proyectos. La elección depende de factores como la complejidad del sistema, el tiempo disponible, el tamaño del equipo y el nivel de participación de los usuarios. Mientras algunos enfoques siguen una secuencia estricta de actividades, otros ofrecen mayor flexibilidad para realizar cambios durante el proceso sin afectar el avance del trabajo.                Uno de los modelos tradicionales más conocidos es el modelo en cascada. Su funcionamiento se basa en desarrollar el software por etapas consecutivas, donde primero se realiza el análisis, luego el diseño, posteriormente la implementación y, al final, las pruebas y el mantenimiento. Esta organización facilita la planificación y la documentación del proceso; sin embargo, cuando aparecen nuevos requerimientos o cambian las necesidades del cliente, volver a una fase ya terminada suele representar más tiempo y un mayor esfuerzo (Pressman y Maxim, 2021).                La necesidad de responder con mayor rapidez a esos cambios dio lugar a metodologías más flexibles, capaces de adaptarse conforme avanza el desarrollo. Entre ellas se encuentra Rapid          Application                   Development          (RAD), un enfoque que propone construir el software mediante ciclos          cortos de trabajo, donde cada versión funcional sirve como base para realizar pruebas, recibir observaciones e incorporar mejoras antes de continuar con la siguiente etapa.                    Desarrollo Rápido de Aplicaciones (RAD)                Rapid          Application                   Development          (RAD) fue propuesto por James Martin como una metodología orientada a reducir los tiempos de construcción mediante el desarrollo iterativo y la participación constante de los usuarios. Su principal diferencia frente a los modelos tradicionales consiste en que no espera a que el producto esté completamente terminado para iniciar las validaciones. Desde las primeras etapas se construyen versiones funcionales que permiten evaluar el comportamiento del sistema y realizar ajustes cuando todavía resulta sencillo corregirlos (Martin, 1991).                El trabajo por iteraciones hace que el proceso sea más dinámico. Cada ciclo incorpora nuevas funcionalidades, corrige errores detectados en versiones anteriores y mejora aquellos aspectos que surgen a partir de la retroalimentación de los usuarios. Esto reduce el riesgo de invertir tiempo en características que luego no aportan valor y permite que la solución evolucione conforme se comprende mejor el problema.                RAD también promueve el uso de prototipos funcionales. En lugar de presentar únicamente documentos o diseños, los usuarios pueden interactuar con una versión operativa del sistema y expresar sus observaciones desde etapas tempranas. Esa participación continua facilita identificar necesidades que inicialmente no habían sido consideradas y mejora la calidad del resultado obtenido.                De forma general, esta metodología comprende cuatro etapas: planificación de requerimientos, diseño con participación del usuario, construcción rápida y puesta en marcha. Aunque existe un orden lógico entre ellas, no se trata de un proceso rígido. Es posible regresar a una etapa anterior cuando aparecen nuevas necesidades o se identifican oportunidades de mejora,          lo que convierte a RAD en una alternativa adecuada para proyectos donde los requerimientos evolucionan durante el desarrollo.                    Ciclo de Vida del Desarrollo de Sistemas (SDLC)                El Ciclo de Vida del Desarrollo de Sistemas, conocido como          System                   Development          Life          Cycle          (SDLC), describe las etapas que normalmente atraviesa un sistema desde que se identifica una necesidad hasta que entra en funcionamiento y recibe mantenimiento. Más que una metodología, constituye un marco de referencia que organiza las actividades necesarias para desarrollar soluciones informáticas de manera estructurada (Pressman y Maxim, 2021).                Aunque existen diferentes interpretaciones del SDLC, la mayoría coincide en etapas como la planificación, el análisis de requerimientos, el diseño, la construcción, las pruebas, la implementación y el mantenimiento. Cada una cumple un propósito específico y proporciona información que sirve de base para la siguiente fase, permitiendo mantener un mayor control sobre el avance del trabajo.                Con el paso del tiempo este modelo ha evolucionado para adaptarse a metodologías más flexibles. Actualmente sus etapas continúan siendo una referencia para organizar el desarrollo, pero ya no necesariamente se ejecutan de forma estrictamente secuencial. En metodologías iterativas es habitual recorrer varias veces algunas fases, incorporando mejoras conforme aparecen nuevos requerimientos o se obtiene retroalimentación de los usuarios.                    Arquitectura de Software y Patrones de Diseño                La arquitectura de software define la forma en que se organizan los componentes de una aplicación y la manera en que interactúan entre sí. Una estructura bien planteada facilita el mantenimiento, mejora la comprensión del código y permite incorporar nuevas funcionalidades sin afectar el comportamiento general del sistema. Cuando cada componente tiene          responsabilidades claramente definidas, el software resulta más sencillo de ampliar y mantener a lo largo del tiempo (Pressman y Maxim, 2021).                Como parte de esa organización también se utilizan patrones de diseño, entendidos como soluciones ampliamente aceptadas para resolver problemas que aparecen con frecuencia durante el desarrollo de software. Estos patrones no corresponden a código listo para utilizar, sino a modelos que orientan la forma de estructurar una solución según las características del problema que se desea resolver (Gamma et al., 1995).                 Entre los patrones arquitectónicos más utilizados se encuentra Modelo-Vista-Controlador (MVC). Este patrón divide la aplicación en tres componentes con responsabilidades específicas: el modelo administra la lógica del negocio y los datos, la vista presenta la información al usuario y el controlador actúa como intermediario entre ambos. Esta separación reduce el acoplamiento entre los diferentes elementos de la aplicación y facilita su mantenimiento conforme el software incorpora nuevas funcionalidades.                              Patrón Modelo-Vista-Controlador (MVC)                A medida que una aplicación incorpora nuevas funciones, mantener el código organizado deja de ser una recomendación y se convierte en una necesidad. Si toda la lógica se concentra en un mismo lugar, cualquier cambio termina afectando diferentes partes del sistema y el mantenimiento se vuelve más complejo. El patrón Modelo-Vista-Controlador (MVC) surge precisamente para evitar ese problema, separando las responsabilidades de la aplicación en componentes independientes que trabajan de forma coordinada (Gamma et al., 1995).                 Dentro de esta arquitectura, el Modelo administra la información y las reglas del negocio. También se encarga de interactuar con la base de datos y representar las entidades que hacen parte del sistema. La Vista corresponde a la interfaz que utiliza el usuario para consultar o          registrar información, mientras que el Controlador recibe las solicitudes, coordina la comunicación entre los demás componentes y determina la respuesta que debe enviarse.                Cuando un usuario realiza una acción, el controlador interpreta la petición y consulta el modelo si necesita acceder o modificar información. Una vez procesados los datos, el resultado se envía a la vista para que sea presentado de forma adecuada. Esta separación evita mezclar la lógica del negocio con la interfaz gráfica y facilita que cada componente pueda modificarse sin afectar el funcionamiento del resto de la aplicación.                El patrón MVC también favorece el trabajo colaborativo, ya que diferentes integrantes del equipo pueden intervenir sobre distintas partes de la aplicación sin generar conflictos constantes en el código. Por esa razón continúa siendo una de las arquitecturas más utilizadas en el desarrollo de aplicaciones web modernas.                    Mapeo Objeto-Relacional (ORM)                El Mapeo Objeto-Relacional, conocido por sus siglas ORM (           Object-Relational                       Mapping         ), es una técnica que establece una relación entre las clases de un lenguaje de programación y las tablas de una base de datos relacional. Su objetivo consiste en reducir la dependencia de consultas SQL escritas manualmente y facilitar el acceso a la información mediante objetos, haciendo que el código resulte más claro y sencillo de mantener (Fowler, 2002).                Gracias a este enfoque, operaciones como crear, consultar, actualizar o eliminar registros pueden realizarse utilizando métodos propios del lenguaje de programación, mientras el ORM se encarga de traducir esas acciones en instrucciones SQL. Esto disminuye la cantidad de código repetitivo y favorece una estructura más organizada durante el desarrollo.                Otra característica importante es el manejo de relaciones entre entidades. Asociaciones como uno a uno,          uno a muchos o muchos a muchos pueden          representarse mediante objetos sin          necesidad de construir consultas complejas para cada operación. Esta capacidad facilita el trabajo con bases de datos que contienen múltiples tablas relacionadas y mejora la legibilidad del código.                Uno de los ORM más utilizados dentro del ecosistema PHP es          Eloquent         , incorporado de forma nativa en Laravel. Su funcionamiento sigue estos principios y permite administrar la información mediante modelos que representan cada tabla de la base de datos, simplificando las operaciones más frecuentes y manteniendo una estructura coherente durante el desarrollo de la aplicación.                    PHP y Laravel                El desarrollo de aplicaciones web del lado del servidor requiere herramientas capaces de procesar solicitudes, administrar información y aplicar las reglas del negocio antes de enviar una respuesta al usuario. Dentro de este contexto, PHP continúa siendo uno de los lenguajes más utilizados gracias a su madurez, amplia comunidad y compatibilidad con la mayoría de los sistemas gestores de bases de datos. Su evolución durante los últimos años ha permitido construir aplicaciones más robustas, incorporando mejoras relacionadas con el rendimiento, la seguridad y la programación orientada a objetos (Nixon, 2023).                El crecimiento de los proyectos también impulsó el uso de          frameworks          que ayudan a organizar el código y reducen la necesidad de desarrollar desde cero funcionalidades comunes. Entre ellos destaca Laravel, un          framework          de código abierto que adopta el patrón MVC y reúne herramientas para el manejo de rutas, autenticación, validación de formularios, acceso a bases de datos, colas de trabajo y desarrollo de interfaces de programación de aplicaciones (API).                Además de proporcionar una estructura organizada, Laravel integra componentes que facilitan el mantenimiento del software conforme aumenta su tamaño. Su ecosistema incorpora Blade como motor de plantillas para construir interfaces dinámicas,          Eloquent          para el acceso a bases de datos mediante ORM y un conjunto de utilidades orientadas a mejorar la productividad          sin sacrificar la calidad del código. Estas características explican por qué Laravel se ha consolidado como una de las alternativas más utilizadas para el desarrollo de aplicaciones web basadas en PHP.                    Gestión de datos y persistencia                La información pierde gran parte de su utilidad si no puede conservarse de manera segura entre una sesión y otra. Por esta razón, las aplicaciones informáticas utilizan mecanismos de persistencia que permiten almacenar los datos de forma permanente y recuperarlos cuando son necesarios. En la mayoría de los sistemas empresariales esta tarea recae sobre las bases de datos relacionales, debido a su capacidad para organizar grandes volúmenes de información manteniendo relaciones consistentes entre los diferentes registros (         Coronel          y Morris, 2019).                Una adecuada gestión de los datos no consiste únicamente en almacenarlos. También implica garantizar su integridad, evitar duplicidades, controlar las relaciones entre las diferentes entidades y facilitar consultas que respondan con rapidez a las necesidades de la aplicación. Un diseño correcto de la base de datos reduce errores, mejora el rendimiento y simplifica el mantenimiento del software conforme aumenta la cantidad de información administrada.                La persistencia constituye, por tanto, uno de los pilares de cualquier sistema de información. Un modelo de datos bien estructurado permite que la información permanezca disponible, organizada y consistente, independientemente del número de usuarios o del crecimiento que experimente la aplicación con el paso del tiempo.                               Bases de Datos Relacionales (MySQL)                Guardar los datos de forma permanente es una necesidad obvia para cualquier software; de nada sirve que la aplicación gestione clientes, empleados, servicios o reservas si esa información se borra al cerrar el sistema. Las bases de datos relacionales resuelven esto organizando todo en tablas conectadas por llaves primarias y foráneas. Este enfoque no solo evita          la duplicidad de datos, sino que simplifica las consultas y mantiene la información ordenada y consistente (         Coronel          y Morris, 2019).                En este modelo, la integridad referencial es una regla de oro. Su trabajo es asegurar que las relaciones entre tablas sigan siendo válidas y que no queden "         datos fantasma         ". Por ejemplo, si un cliente se elimina, el sistema debe saber qué hacer con sus reservas viejas en lugar de dejar un enlace roto a un usuario inexistente. Configurar bien estas restricciones cuida la calidad de los datos y ahorra muchos dolores de cabeza por errores en plena operación.                Para interactuar con esta estructura se utiliza SQL. Con instrucciones directas como SELECT, INSERT, UPDATE o DELETE se manipulan los registros. Además, cuando hace falta cruzar datos de distintos lados (como saber qué empleado atendió qué cita), operaciones como los JOIN permiten unificar la información de manera limpia. En este ecosistema también aparecen los            triggers          o disparadores: pequeñas rutinas que saltan automáticamente ante un evento (un cambio o una inserción) y que resultan muy útiles para automatizar tareas o validar reglas del negocio desde el propio motor de la base de datos.                Entre las opciones disponibles, MySQL se ha convertido en el estándar de confianza tanto en entornos de aprendizaje como en la industria. Es estable, rápido, fácil de administrar y se integra sin problemas con casi cualquier lenguaje o            framework         , lo que lo hace ideal para proyectos web que buscan un manejo de datos seguro y eficiente.                             Versionamiento            de Base de Datos (Migraciones y Seeders)                La estructura de una base de datos nunca es estática. A medida que el software evoluciona y surgen nuevos requerimientos, el diseño inicial cambia: se crean tablas, se añaden columnas, se ajustan relaciones o se descartan campos obsoletos. Hacer estas modificaciones de forma manual (directamente en el gestor de base de datos) es una práctica de alto riesgo, ya que rompe la          sincronía entre los entornos de desarrollo, prueba y producción, volviéndose un dolor de cabeza cuando varios programadores trabajan en el mismo proyecto.                Para solucionar este caos, Laravel implementa las migraciones, que funcionan esencialmente como un sistema de control de versiones —un Git— pero exclusivo para la base de datos. Estos archivos describen la estructura del esquema mediante código PHP. Gracias a este enfoque, cualquier cambio en las tablas se puede replicar de manera idéntica y controlada en la computadora de cualquier miembro del equipo o en el servidor final, garantizando que todos compartan exactamente el mismo modelo de datos (Stauffer, 2023).                A la par de las migraciones, el            framework          ofrece los            seeders          o sembradores, diseñados para poblar esas tablas con información inicial o registros de prueba. En lugar de registrar filas a mano cada vez que la base de datos se limpia o se despliega desde cero, los            seeders          inyectan datos consistentes de manera automática. Esto agiliza las pruebas de la aplicación, permitiendo evaluar el comportamiento del sistema (como el inicio de sesión o el flujo de una cita) de forma inmediata.                El uso combinado de migraciones y            seeders          no solo aporta orden y trazabilidad al historial del proyecto, sino que elimina los típicos errores provocados por diferencias entre entornos de trabajo. Además, simplifica drásticamente la portabilidad del software, logrando que un nuevo desarrollador pueda levantar todo el ecosistema de datos con un par de comandos en la terminal.                    Seguridad Web                Cuando un sistema gestiona datos sensibles como registros de clientes, jornadas de empleados, agendas y transacciones financieras, la seguridad deja de ser una opción y se convierte en un pilar del desarrollo. Una aplicación no está terminada solo porque "funcione"; también debe garantizar que la información esté blindada contra accesos no autorizados o ataques          que alteren su comportamiento. Para lograrlo, el diseño de la arquitectura debe responder activamente a los tres principios fundamentales de la seguridad de la información:           confidencialidad, integridad y disponibilidad         (Stallings, 2021).                Abordar este desafío requiere anticiparse a los riesgos desde la etapa de diseño. Prácticas esenciales como la validación estricta de las peticiones de los usuarios, el control estricto de permisos y la implementación de esquemas robustos de autenticación reducen drásticamente la superficie de ataque del software, protegiendo los datos que se guardan de forma permanente.                    Anatomía de las Amenazas Web Comunes                En el ecosistema web, existen vectores de ataque clásicos que la fundación OWASP sigue catalogando como críticos debido a su frecuencia e impacto (OWASP          Foundation         , 2021). Comprender cómo operan es vital para neutralizarlos:                      Inyección SQL (            SQL              Injection           ):         Ocurre cuando datos maliciosos enviados por un usuario son interpretados de forma errónea como comandos por el motor de la base de datos. Si el sistema no filtra estos campos, un atacante podría secuestrar las consultas, logrando leer, modificar o destruir tablas completas.                      Cross-Site Scripting (XSS):         Este ataque introduce código malicioso (generalmente JavaScript) en las páginas web que ven otros usuarios. Si la aplicación imprime datos en la vista sin antes "escapar" o limpiar el texto, el navegador de la víctima ejecutará el código, abriendo la puerta al robo de cookies de sesión o a la manipulación visual de la interfaz.                      Cross-Site            Request                       Forgery            (CSRF):         Esta técnica aprovecha la confianza que el sitio web ya tiene en un usuario autenticado. Mediante engaños externos, el atacante induce al navegador de la víctima a realizar solicitudes dañinas (como cambiar contraseñas o realizar registros) mientras su sesión legítima sigue activa en segundo plano.                                 Arquitectura de Servicios Web: API REST                El software moderno ya no se concibe como una isla monolítica. Hoy en día, es indispensable que una aplicación web pueda comunicarse de manera fluida con aplicaciones móviles, pasarelas de pago, servicios externos o plataformas de terceros. Las Interfaces de Programación de Aplicaciones (API) resuelven esta necesidad, actuando como un puente tecnológico que establece reglas claras para el intercambio de datos entre sistemas construidos con tecnologías totalmente diferentes.                El estándar de oro para diseñar estas interfaces es el estilo arquitectónico REST (           Representational            State Transfer        ), acuñado originalmente por Roy Fielding. Bajo este enfoque, cualquier entidad de la aplicación (como un cliente, una reserva o un servicio) se trata como un "recurso" único identificado mediante una dirección URL. La manipulación de estos recursos se realiza aprovechando los verbos nativos del protocolo HTTP: GET para leer, POST para crear, PUT o PATCH para modificar y DELETE para eliminar (Fielding, 2000).                La genialidad de REST radica en su simplicidad y en el uso de estándares preexistentes en la web, lo que elimina la necesidad de capas complejas de mensajería y favorece la escalabilidad del sistema. Para el transporte de los datos, el formato predilecto es JSON (          JavaScript            Object                       Notation         ). Al ser una estructura de texto ligera, basada en pares clave-valor, es sumamente fácil de procesar por el servidor y de interpretar por cualquier cliente web o móvil, consolidándose como la convención universal para la integración de aplicaciones.                    Virtualización Ligera con Docker y Contenedores                Históricamente, el despliegue de software ha sufrido por el clásico dilema:           "En mi computadora sí funciona, pero en el servidor no"        . Las sutiles diferencias entre sistemas operativos, versiones de PHP, extensiones faltantes o configuraciones de bases de datos solían          romper las aplicaciones al cambiar de entorno. Los contenedores llegaron para erradicar este problema de raíz.                Un contenedor es un entorno aislado que empaqueta la aplicación junto con todo lo que necesita para ejecutar de forma correcta (código, binarios, librerías y dependencias). A diferencia de las máquinas virtuales tradicionales, que imitan un hardware completo y cargan un sistema operativo entero sobre otro, los contenedores comparten el mismo núcleo (           kernel         ) del sistema operativo anfitrión. Esto los hace increíblemente ligeros, rápidos de arrancar y sumamente eficientes en el consumo de recursos (Turnbull, 2024).                En este escenario, Docker se ha consolidado como la herramienta líder para la gestión de contenedores. Mediante archivos de configuración de infraestructura como código, Docker permite definir entornos de desarrollo idénticos y reproducibles para todo el equipo. Además, facilita la arquitectura de microservicios o servicios independientes; por ejemplo, permite que el servidor web, el motor de MySQL y las herramientas de administración coexistan en contenedores separados, comunicándose entre sí de forma limpia y sin interferir con el sistema operativo local del programador.                                           Marco Contextual                 El presente proyecto se desarrolló tomando como caso de estudio la Barbería y Perfumería          JyM         , establecimiento dedicado a la prestación de servicios de barbería ubicado en el municipio de Cartago, departamento del Valle del Cauca. La investigación se delimitó geográficamente a este municipio y sector económico, con el propósito de analizar una necesidad real y plantear una solución tecnológica ajustada a las características del negocio.                 Cartago se encuentra localizado al norte del departamento del Valle del Cauca y constituye uno de los principales centros urbanos de esta subregión. Su ubicación estratégica facilita la conexión con departamentos como Risaralda y Quindío, favoreciendo la actividad comercial y la prestación de servicios. De acuerdo con las proyecciones poblacionales del Departamento Administrativo Nacional de Estadística (DANE, 2026), el municipio cuenta actualmente con una población de 144.206 habitantes. Su economía se sustenta principalmente en el comercio, la prestación de servicios, la pequeña industria y las actividades agropecuarias. Estas características demográficas y comerciales han impulsado el crecimiento de numerosos establecimientos orientados a la atención directa del público, entre ellos las barberías.                En los últimos años, el sector de las barberías ha experimentado un crecimiento importante en el territorio nacional, dejando de ser una actividad cotidiana e informal para convertirse en una industria robusta, sofisticada y con una amplia cadena de valor (Cámara de Comercio de Armenia y del Quindío, 2026). Este auge está fuertemente impulsado por la mayor demanda de servicios relacionados con el cuidado de la imagen masculina y el aseo personal, un mercado que a nivel nacional registra incrementos anuales sostenidos en el gasto per cápita de los usuarios (Asociación Nacional de Empresarios de Colombia [ANDI], 2024). Además del corte de cabello tradicional, muchos establecimientos ofrecen servicios complementarios como arreglo de barba, tratamientos capilares, venta de productos para el cuidado personal y asesoría en estilos. Este aumento en la oferta también ha incrementado la necesidad de mejorar la organización interna de los negocios para brindar una atención más eficiente y reducir errores durante la programación de citas.                La Barbería y Perfumería          JyM          desarrolla sus actividades mediante la atención de clientes previamente agendados y la prestación de diferentes servicios de barbería. Antes del desarrollo del presente proyecto, la programación de citas se realizaba principalmente mediante          conversaciones por WhatsApp y registros manuales, mientras que la información relacionada con clientes, empleados y servicios permanecía distribuida en diferentes medios. Esta forma de trabajo —basada en herramientas de mensajería instantánea no especializadas— suele dificultar el seguimiento de las reservas, la consulta del historial de atención y genera reprocesos administrativos por la falta de centralización de los datos (         Koalendar         , 2025).                Frente a esta situación, el proyecto se enfocó en intervenir el proceso de gestión de reservas y administración de la información del negocio. Para ello se diseñó una aplicación web capaz de centralizar la gestión de clientes, empleados, servicios, productos, horarios y reservas dentro de un único sistema. La solución también incorpora mecanismos para controlar la disponibilidad de los empleados, registrar el historial de los servicios prestados y facilitar la administración de la información necesaria para el funcionamiento del establecimiento.                Los usuarios contemplados dentro del sistema corresponden principalmente al personal administrativo de la barbería, encargado de gestionar la información del negocio, registrar clientes, administrar empleados, controlar el catálogo de servicios y productos, organizar las reservas y consultar la información almacenada. De igual forma, el sistema administra los datos relacionados con los clientes que solicitan los servicios del establecimiento, permitiendo conservar el historial de las reservas realizadas.                En consecuencia, el alcance del proyecto se limita al desarrollo de una aplicación web orientada exclusivamente a apoyar los procesos administrativos de la Barbería y Perfumería          JyM          en el municipio de Cartago. La solución busca mejorar la organización de la información, optimizar la programación de reservas y fortalecer el control de las operaciones internas del establecimiento, sin contemplar en esta etapa funcionalidades dirigidas a otros sectores económicos ni procesos diferentes a los propios de una barbería.                     Capítulo I        : Requerimientos del Sistema                   Introducción del capítulo                 El análisis de requerimientos constituye la fase inicial y más determinante del ciclo de vida de cualquier sistema de información. Su propósito es identificar, documentar y validar las necesidades que el sistema debe satisfacer, tanto desde la perspectiva de las funcionalidades requeridas como desde los atributos de calidad que debe cumplir durante su operación. De acuerdo con Pressman (2010), comprender los requerimientos de un problema es condición necesaria para alcanzar el diseño de una solución eficaz; abordar esta fase de manera superficial es una de las causas más frecuentes de fracaso en los proyectos de software.                         El presente capítulo recoge el producto del análisis de requerimientos realizado para el sistema PYME Reservas, una aplicación web orientada a la gestión de servicios y reservas de la Barbería y Perfumería          JyM         , establecimiento ubicado en el municipio de Cartago, Valle del Cauca. El levantamiento de información se llevó a cabo en coherencia con la metodología Desarrollo Rápido de Aplicaciones (RAD), la cual propone ciclos iterativos que permiten refinar progresivamente los requerimientos con base en la retroalimentación obtenida en cada etapa de desarrollo. Esta aproximación difiere del modelo en cascada, donde los requerimientos se congelan antes de iniciar la construcción; en RAD, la especificación y el desarrollo ocurren de manera paralela y se retroalimentan mutuamente (Martin, 1991).                         Para la documentación de los requerimientos funcionales se adoptó la estructura propuesta por el Instituto de Ingenieros Eléctricos y Electrónicos (IEEE, 1998), que establece la completitud, consistencia, verificabilidad y trazabilidad como atributos fundamentales de una especificación de calidad. Sommerville (2016) complementa este enfoque al distinguir entre requerimientos de usuario que describen en lenguaje natural lo que el sistema debe hacer y requerimientos del sistema que especifican con mayor precisión las funciones, restricciones y          comportamientos esperados—. Ambos niveles de especificación se encuentran representados en este capítulo.                         El capítulo se organiza de la siguiente manera: en primer lugar, se identifican los actores del sistema y se describen sus responsabilidades dentro del dominio del negocio. A continuación, se presentan los requerimientos funcionales y no funcionales que surgieron del análisis realizado. Posteriormente, se establecen las reglas de negocio que gobiernan el comportamiento del sistema y se describen los casos de uso identificados, acompañados de su correspondiente matriz y de las instrucciones para la elaboración del diagrama de casos de uso. El capítulo concluye con la delimitación precisa del alcance funcional y una declaración explícita de las exclusiones del sistema, con el fin de establecer expectativas claras sobre lo que la solución actual comprende y lo que queda fuera de su alcance.          Actores del sistema                         Un actor, en el contexto del análisis de sistemas de información, representa cualquier entidad externa que interactúa con el sistema para obtener o proveer información, desencadenar procesos o consumir resultados (Jacobson et al., 1999). Los actores no forman parte del sistema en sí, pero su identificación precisa es condición necesaria para delimitar correctamente las funcionalidades requeridas y las restricciones de acceso que el sistema debe aplicar.                        PYME Reservas reconoce tres actores principales, cada uno con un conjunto de responsabilidades diferenciadas y un nivel de privilegio distinto dentro del sistema. El control de acceso basado en roles (RBAC, por sus siglas en inglés) garantiza que cada actor únicamente pueda ejecutar las operaciones que le corresponden según su función en el negocio.          Administrador                         El Administrador es el actor con mayor nivel de privilegio dentro del sistema. Corresponde al propietario del establecimiento o a la persona designada para gestionar la          operación completa de la barbería. Tiene acceso irrestricto a todos los módulos del sistema y es el único actor autorizado para modificar la configuración general, gestionar usuarios y consultar los indicadores de desempeño del negocio. Sus responsabilidades dentro del sistema comprenden:                    Crear, consultar, editar y eliminar registros de usuarios, empleados y clientes.                    Configurar los servicios ofrecidos por el establecimiento, incluyendo nombre, precio, duración, imagen de referencia y estado de disponibilidad.                    Administrar el catálogo de productos, controlando el precio, el stock disponible, las imágenes y el estado de cada referencia.                    Gestionar el ciclo de vida completo de las reservas: creación, edición, confirmación, cancelación, marcado como completada y registro de no asistencia.                    Definir y mantener los horarios laborales de cada empleado, incluyendo los días de descanso y las franjas horarias de atención.                    Consultar el panel de control administrativo, que consolida ingresos del período, indicadores clave de desempeño (         KPIs         ), servicios más solicitados, barberos con mayor productividad, estado del inventario y resumen de reservas.          Empleado (Barbero)                         El Empleado representa al personal de servicio del establecimiento. Su acceso al sistema está restringido exclusivamente a la información relacionada con su propia actividad laboral. Esta restricción no es únicamente una decisión de diseño, sino una regla de negocio explícita: un empleado no puede visualizar, modificar ni interferir en las reservas asignadas a otro compañero ni en la configuración del sistema. Sus responsabilidades dentro del sistema comprenden:                    Consultar su agenda personal con las citas asignadas para el día, la semana o el período que requiera visualizar.                     Revisar el detalle de cada reserva asignada: cliente, servicios solicitados, hora de inicio, hora de finalización estimada y estado actual de la cita.                    Actualizar el estado de las reservas a su cargo, marcándolas como completadas o registrando la no asistencia del cliente cuando corresponda.                        El sistema impide al Empleado acceder a información financiera, registros de otros empleados, configuración de servicios o cualquier módulo de administración. Este control se aplica mediante middleware y políticas de autorización que validan el rol del usuario autenticado en cada solicitud.          Cliente                         El Cliente es el actor externo que consume los servicios del establecimiento. Interactúa con el sistema a través del portal de cliente, una interfaz diferenciada que le permite gestionar su relación con la barbería de forma autónoma, sin requerir la intervención del Administrador para operaciones básicas. Sus responsabilidades dentro del sistema comprenden:                    Registrarse en el sistema proporcionando sus datos de contacto y creando sus credenciales de acceso.                    Iniciar sesión para acceder a las funcionalidades del portal.                    Consultar el catálogo de servicios disponibles, con sus descripciones, precios y duraciones.                    Crear reservas seleccionando el servicio o servicios de su interés, el empleado de su preferencia, la fecha y la franja horaria disponible.                    Consultar el historial y el estado de sus reservas activas y pasadas.                    Cancelar reservas futuras desde su perfil, dentro de los límites que las reglas de negocio establecen.                    Actualizar su información de perfil, incluyendo datos de contacto y contraseña.                         El cliente no tiene acceso a información de otros clientes, a registros de empleados ni a ningún módulo de administración. La protección contra manipulación de identificadores en las URL (IDOR, por sus siglas en inglés) garantiza que un cliente autenticado únicamente pueda operar sobre sus propios datos.          Requerimientos funcionales                                 Los requerimientos funcionales describen el conjunto de funcionalidades que el sistema debe ofrecer para satisfacer las necesidades identificadas durante el análisis del negocio. Cada requerimiento representa una acción o servicio que la aplicación debe ejecutar en respuesta a las interacciones de los diferentes actores del sistema.                Para facilitar su organización, identificación y trazabilidad durante el desarrollo, cada requerimiento se documenta mediante un código único, un nombre descriptivo, una descripción funcional y el actor responsable de su ejecución. Esta estructura permite relacionar posteriormente cada requerimiento con los casos de uso, los módulos implementados y las funcionalidades desarrolladas en la aplicación.                Los requerimientos se encuentran organizados por módulos funcionales, siguiendo la arquitectura implementada en el sistema, lo que facilita la comprensión del alcance de cada componente y la verificación de que todas las funcionalidades previstas fueron contempladas durante el proceso de desarrollo. RN-05 Privacidad de Agenda por Barbero:      Un empleado no puede andar mirando quién va a motilarse con sus compañeros ni meterse a alterar las reservas de los demás. Capítulo II        : Diseño de la Solución Web       El presente capítulo detalla el diseño arquitectónico e implementación de la solución web para la Barbería y Perfumería      JyM     . Arquitectura del Sistema y Tecnologías Empleadas       L    a solución propuesta fue desarrollada bajo una arquitectura Cliente-Servidor, implementando el patrón de diseño Modelo-Vista-Controlador (MVC), el cual permite separar la lógica del negocio, la interfaz de usuario y el acceso a los datos. Esta organización facilita el mantenimiento del sistema, mejora la reutilización del código y permite realizar futuras modificaciones sin afectar el funcionamiento general de la aplicación.      Para el desarrollo del sistema se emplearon tecnologías actuales que ofrecen estabilidad, seguridad y un buen rendimiento:             Backend     : Se utilizó Laravel 13 como      framework      de desarrollo sobre PHP 8.4, aprovechando las herramientas que ofrece el      framework      para la gestión de rutas, autenticación, validación de datos y acceso a la base de datos mediante el ORM      Eloquent     .              Frontend     : La interfaz fue construida utilizando el motor de plantillas Blade, complementado con      Tailwind      CSS para el diseño adaptable a diferentes dispositivos y JavaScript para implementar funcionalidades dinámicas, como la consulta de disponibilidad de horarios y la actualización de información sin necesidad de recargar la página.             Base de datos: Se empleó MySQL 8 como sistema gestor de bases de datos relacionales, encargado del almacenamiento y administración de toda la información del sistema.             Entorno de desarrollo: Para garantizar un ambiente de trabajo uniforme durante el desarrollo y facilitar futuras implementaciones, el proyecto se ejecutó mediante Docker, utilizando Laravel      Sail      para la administración de los contenedores y sus servicios.             Gestión de la información y base de datos       La información del sistema se almacena en una base de datos relacional, diseñada siguiendo principios de normalización con el propósito de minimizar la redundancia de datos y mantener la integridad de la información.      La estructura de la base de datos permite administrar los diferentes procesos de la barbería y perfumería, incluyendo la gestión de usuarios, clientes, empleados, servicios, productos, reservas, horarios de trabajo y pedidos realizados por los clientes.      Asimismo, el modelo implementa llaves foráneas y restricciones de integridad que garantizan la consistencia de los datos, evitando situaciones como asignar una reserva a un empleado inexistente, registrar servicios inválidos o generar pedidos con información inconsistente. Esta organización proporciona una base sólida para el correcto funcionamiento del sistema y facilita su crecimiento en futuras versiones. En segundo lugar, el sistema implementa un modelo de control de acceso basado en roles y permisos. Finalmente, para las operaciones críticas del negocio, como el registro de reservas y la confirmación de pedidos, se emplean transacciones de base de datos. Esto permite que todas las operaciones relacionadas se ejecuten como una única unidad de trabajo; si ocurre algún error durante el proceso, los cambios realizados se revierten automáticamente, preservando la integridad de la información y evitando inconsistencias en la base de datos. Esta separación entre la interfaz web y la capa de servicios permite que el sistema pueda evolucionar fácilmente hacia aplicaciones móviles o integraciones con otros sistemas sin modificar la lógica principal del negocio. Ilustración         SEQ Ilustración \* ARABIC           6        Diagrama de Secuencia                                                                    Capítulo III        : Mecanismos de Seguridad para el Funcionamiento de la Aplicación           Seguridad y Permisos de los Usuarios                  Uno de los aspectos más importantes durante el desarrollo del sistema fue garantizar que cada usuario pudiera acceder únicamente a la información y las funciones que le corresponden. Para lograrlo, se implementó un esquema de control de acceso basado en roles (RBAC), el cual permite asignar diferentes niveles de permisos según el tipo de usuario.                 En el sistema existen tres roles principales:           Administrador        ,           Empleado (Barbero)         y           Cliente        . El administrador cuenta con acceso completo a los módulos de gestión, incluyendo usuarios, empleados, clientes, servicios, productos, inventario y reservas. Los empleados únicamente pueden consultar y administrar la información relacionada con sus propias citas y          actividades, mientras que los clientes solo tienen acceso a sus reservas, pedidos e información personal.                 Adicionalmente, se aprovecharon los mecanismos de seguridad que ofrece Laravel, como la protección contra ataques de inyección SQL, Cross-Site Scripting (XSS) y Cross-Site          Request                   Forgery          (CSRF), contribuyendo a mantener la integridad y confidencialidad de la información almacenada en la plataforma.          Pruebas del Sistema                  Con el fin de garantizar el correcto funcionamiento de la aplicación, se realizaron pruebas sobre los principales procesos del sistema, especialmente aquellos relacionados con el agendamiento de citas, la autenticación de usuarios y la gestión de inventario.                 Estas pruebas permitieron verificar que las reglas de negocio se cumplieran correctamente, evitando situaciones como la asignación de dos citas en el mismo horario para un mismo barbero, el acceso no autorizado a información de otros usuarios o inconsistencias en el control del inventario.                 El objetivo de estas validaciones fue asegurar que el sistema respondiera de manera confiable ante diferentes escenarios de uso y ofreciera estabilidad durante su funcionamiento.          Entorno de Desarrollo con Docker                  Para facilitar el desarrollo del proyecto y garantizar que todos los integrantes del equipo trabajaran bajo las mismas condiciones, se utilizó Docker junto con Laravel          Sail         .                 Esta herramienta permitió ejecutar el servidor web, la base de datos y los demás servicios necesarios dentro de contenedores independientes, evitando problemas de compatibilidad entre diferentes equipos o sistemas operativos.                  Gracias a esta estrategia, el proyecto puede desplegarse fácilmente en otro computador o servidor conservando la misma configuración utilizada durante el desarrollo, reduciendo errores de instalación y mejorando la portabilidad de la aplicación. Recomendaciones                  Se recomienda implementar el sistema de manera gradual, iniciando con el módulo de reservas y posteriormente incorporando el control de inventario y la gestión de pedidos, permitiendo que el personal se adapte progresivamente al uso de la plataforma.                 Es importante realizar copias de seguridad periódicas de la base de datos con el fin de proteger la información de clientes, reservas, productos e historial de ventas frente a posibles fallos técnicos o pérdidas de información.                 Asimismo, se recomienda capacitar al personal administrativo y a los barberos en el manejo del sistema para garantizar un uso adecuado de las diferentes funcionalidades y aprovechar al máximo las herramientas implementadas.                 Finalmente, se aconseja mantener actualizados tanto el servidor como las dependencias del proyecto, con el propósito de conservar un ambiente seguro, estable y compatible con futuras versiones del software.                  Recomendaciones técnicas                 Como trabajo futuro, se recomienda implementar un sistema de notificaciones automáticas mediante correo electrónico o WhatsApp para recordar a los clientes sus citas y reducir la inasistencia.                 También sería conveniente integrar una pasarela de pagos en línea que permita realizar el pago de las reservas y pedidos antes de asistir al establecimiento, facilitando el proceso tanto para los clientes como para la administración.                  Otra mejora importante consiste en desarrollar reportes estadísticos más avanzados que permitan analizar indicadores como los servicios más solicitados, horarios de mayor demanda, desempeño de los empleados y comportamiento de las ventas.                 Igualmente, se recomienda fortalecer el módulo de auditoría registrando las acciones realizadas por los usuarios, con el fin de llevar un mayor control sobre las operaciones ejecutadas dentro del sistema.                  Recomendaciones para futuros proyectos      Como una posible evolución del sistema, se recomienda desarrollar una aplicación móvil para dispositivos Android      e      iOS que permita a los clientes gestionar sus reservas y pedidos desde cualquier lugar.      También se podría implementar un sistema de fidelización mediante acumulación de puntos, promociones personalizadas y descuentos para clientes frecuentes.      Finalmente, se recomienda integrar el sistema con herramientas de facturación electrónica, plataformas contables y servicios de mensajería, permitiendo ampliar las funcionalidades de la solución y facilitar su adopción en otros establecimientos del sector.
